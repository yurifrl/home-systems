A pitch — draft

nostos / bring your bare metal home to the cluster

One CLI. Plug in a machine. Get a cluster. Keep your sanity.

νόστος · Homer's word for homecoming. The Odyssey is one long nostos. So is every bare-metal install.

The story so far

Getting a Talos node onto your home lab takes, today, eight hours of a very patient evening. I know — I just did it. Here's what it required on the last attempt:

  • Four shell scripts with hand-written dnsmasq invocations
  • Four taskfiles/ of overlapping commands, some pointing at a decommissioned Raspberry Pi
  • A machineconfig YAML template rendered by op inject into a second directory
  • Manual regeneration of talosconfig when the admin cert expires (silently)
  • BIOS F12 dance to convince the Dell to PXE-boot over its own disk
  • A talos.experimental.wipe=system kernel arg that, if left in, loops forever
  • Paper-taped knowledge of MAC addresses, iPXE ${next-server} vs ${filename} race conditions, and why Deco routers like to win the DHCP race

None of this is wrong. It all works. But every single step is undocumented tribal knowledge. Re-running in six months means re-reading a 900-line conversation log.

nostos is the tool I wish existed when I started.

What nostos is

A single Python CLI (and a small web UI) that owns the entire bare-metal onboarding loop:

nostos node add dell01                 # interactive: MAC, IP, role, disk
nostos build                          # download Talos + build iPXE, once
nostos render dell01                   # op-inject secrets → per-MAC config
nostos serve                          # start PXE (dnsmasq + HTTP)
# → power on the Dell, walk away
nostos bootstrap dell01               # talosctl bootstrap + wait for Ready

Five commands. That's the happy path.

Architecture

                        ┌──────────────────────┐
                        │  nostos/templates/*.yaml │  source of truth (committed)
                        └──────────┬───────────┘
                                   │
                                   │ resolve via secrets-backend (op/sops/env/file)
                                   ▼
                        ┌──────────────────────┐
                        │ nostos/state/configs/ │  per-MAC rendered (gitignored)
                        └──────────┬───────────┘
                                   │
 ┌──────────┐   DHCP/TFTP   ┌──────┴───────┐   HTTP    ┌─────────────┐
 │  dnsmasq │◀─────────────▶│ nostos serve │◀─────────▶│ Talos node  │
 │  (sudo)  │               │  (python)    │           │ (PXE boot)  │
 └──────────┘               └──────┬───────┘           └─────────────┘
                                   │
                                   │ talosctl API (v1alpha1)
                                   ▼
                        ┌──────────────────────┐
                        │ nostos bootstrap /   │  post-install
                        │ status / config refresh  │
                        └──────────────────────┘

Three layers:

  • Template layer — human-written YAML with op:// refs. Committed. Reviewable in PRs.
  • Serving layer — dnsmasq + HTTP + rendered configs. Ephemeral, regenerable, gitignored.
  • Control layer — talosctl wrappers for bootstrap, status, rotate, wipe.

CLI surface

CommandWhat it doesReplaces
nostos init Scaffold nostos/config.yaml, templates/, nostos/state/, .gitignore manual setup
nostos node add <name> Interactive wizard: MAC, IP, role, disk, template. Writes nostos/config.yaml + template stub. hand-edit pxe/nodes.yaml
nostos node list Table of nodes with live reachability, Talos version, role, Ready status (didn't exist)
nostos build Download Talos kernel/initramfs from factory.talos.dev; build iPXE snponly.efi with retry-loop embed task pxe:setup
nostos render <node> Resolve secrets (op, sops, env, file), render per-MAC config, talosctl validate task pxe:config
nostos serve Start HTTP + dnsmasq in foreground; clear stale state; live log with request filter task pxe:up
nostos install <node> Print exact boot sequence for that node: BIOS keys, PXE flow, expected outcome tribal knowledge
nostos wipe <node> One-shot PXE boot with wipe=system; auto-removes flag after install completes manual edit + hope
nostos bootstrap <node> talosctl bootstrap; wait for etcd + apiserver; fetch kubeconfig manual
nostos config refresh Re-generate admin cert against the CA (handles the "cert expired" trap) /tmp/talos-ca dance
nostos status Cluster health + per-node STAGE / READY / version / ext-services talosctl dashboard × N
nostos web Start localhost UI on :8080 —

Secrets: 1Password stays, but it's just an adapter

The op://kubernetes/talos/TS_AUTHKEY pattern is good. Keep it. But treat it as one backend behind a common interface:

# nostos/config.yaml
secrets:
  backend: onepassword        # onepassword | sops | env | file
  onepassword:
    account: my.1password.com
    vault: kubernetes

A future move to sops (encrypted-in-repo) or shared team vault is a 20-line plugin, not a rewrite. Template files stay identical — op:// refs become sops:// or env:// with one flag.

What this is not

  • Not Sidero Omni. Omni is the Talos vendor's hosted SaaS. nostos is local-only, offline-capable, 0 RPS of phone-home.
  • Not Matchbox / Tinkerbell / MaaS. Those are production datacenter provisioners. nostos is one person, one laptop, one weekend.
  • Not a Kubernetes bootstrapper. Talos already does that. nostos just gets Talos onto metal.
  • Not opinionated about your network. If your router runs DHCP, nostos's dnsmasq co-exists via vendor-class filtering (we already do this).

Why open-source

Every Talos home-labber hits the same eight walls. The Talos docs are excellent but assume you've done metal provisioning before. Every blog post about "my home Kubernetes" reinvents the same 4 shell scripts we just wrote. There should be a single tool that captures the hard-won knowledge once.

Small surface area (Python, uv, single binary via uv run --script). Easy to read. Easy to fork. MIT. Ships with the real troubleshooting for real failure modes: Deco DHCP races, ${next-server} vs ${filename}, persisted STATE partition, expired admin certs. The things that actually eat weekends.

Migration plan (for this repo)

  1. v0.1: port pxe/scripts/*.sh verbatim into nostos subcommands. No feature changes. Prove parity.
  2. v0.2: add nostos config refresh (the /tmp/talos-ca dance), nostos bootstrap, nostos status. Delete taskfiles/pxe.yml.
  3. v0.3: nostos node add wizard; nostos/config.yaml becomes source of truth; pxe/nodes.yaml retired.
  4. v0.4: nostos web. Delete taskfiles/talostroobleshooting.yml.
  5. v1.0: extract from this repo to its own. Publish on PyPI / brew. Write a proper README with a GIF of a node going from "just unboxed" to "Ready" in one command.

Open questions

  • Single-file uv run --script vs. proper package layout? Probably script at first (ship-ability), package once web UI lands.
  • Ship a pre-built iPXE for common arches, or always build locally via Docker? Shipping avoids the Docker dep for 95% of users.
  • Support metal + nocloud only, or also aws/azure? Start metal-only. Scope creep kills.
  • Web UI mutates state (restart, wipe)? Or read-only? Lean read-only + "copy this command" buttons. Safer, less code.
